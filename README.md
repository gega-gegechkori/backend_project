# TechnicShop - E-Commerce Backend API
# გეგა გეეგჭკორი, ანდრო ძამუკაშვილი, ირაკლი ჯოჯუა

## პროექტის შესახებ

**TechnicShop** არის ელექტრონული კომერციის (E-commerce) პლატფორმის ბექენდ აპლიკაცია, რომელიც სპეციალიზებულია ტექნიკის ონლაინ მაღაზიის მართვაზე. პროექტი აგებულია **Spring Boot** ფრეიმვორქის ბაზაზე და მიყვება **RESTful API** არქიტექტურას.

აპლიკაცია უზრუნველყოფს სწრაფ, მასშტაბირებად და დაცულ სერვისებს ონლაინ ვაჭრობის სრული ციკლისთვის — მომხმარებლის რეგისტრაციიდან და ავტორიზაციიდან, საყიდლების კალათის მართვის, დინამიური ფასდაკლებების, პრომო კოდებისა და შეკვეთის საბოლოო გაფორმებამდე.

## ტექნოლოგიური სტეკი (Tech Stack)

* **Java (JDK 17+)** — ძირითადი პროგრამირების ენა.
* **Spring Boot (Web, Data JPA, Security)** — აპლიკაციის არქიტექტურის, მონაცემთა ბაზასთან ურთიერთქმედებისა და უსაფრთხოების საფუძველი.
* **Spring Security & JWT (JSON Web Tokens)** — უსაფრთხო, Stateless ავტორიზაცია როლებზე დაფუძნებული წვდომის კონტროლით (RBAC).
* **Hibernate (ORM)** — ობიექტურ-რელაციური ასახვა მონაცემთა ბაზის ცხრილებთან.
* **Maven** — პროექტის დამოკიდებულებებისა და ბილდის მართვა.
* **Relational Database** — თავსებადია ნებისმიერ რელაციურ ბაზასთან (მაგ. PostgreSQL, MySQL).

## პროექტის არქიტექტურა და კოდის სტრუქტურა

პროექტი მიყვება ფენოვან არქიტექტურას (Layered Architecture), რაც უზრუნველყოფს კოდის ფუნქციონალურ განცალკევებას:

* **controllers**: HTTP მოთხოვნების მიმღები ფენა, სადაც განსაზღვრულია API მარშრუტები (Endpoints).
* **services**: ბიზნეს ლოგიკის ძირითადი ფენა, სადაც ხდება მონაცემების დამუშავება, ფასდაკლებების გამოთვლა და ვალიდაციები.
* **repositories**: მონაცემთა ბაზასთან ურთიერთქმედების ფენა Spring Data JPA-ის ბაზაზე.
* **entities**: მონაცემთა ბაზის ORM მოდელები (User, Product, Cart, Order და ა.შ.).
* **dto (Data Transfer Objects)**: კლასები კლიენტსა და სერვერს შორის მონაცემების ოპტიმალურად მიმოცვლისთვის.

### შეცდომების გლობალური დამუშავება (Global Exception Handling)
აპლიკაციაში რეალიზებულია შეცდომების ცენტრალიზებული დამუშავება `BaseController` კლასში `@ControllerAdvice` ანოტაციის გამოყენებით. სისტემაში გათვალისწინებულია შემდეგი სცენარები:
* **AccessDeniedException (403 Forbidden)**: ბრუნდება ტექსტი `"forbidden_access"`, როდესაც მომხმარებელს არ აქვს კონკრეტული მოქმედების უფლება.
* **ResourceNotFoundException & UsernameNotFoundException (404 Not Found)**: ბრუნდება მაშინ, როდესაც ბაზაში ვერ მოიძებნა სასურველი პროდუქტი, კატეგორია ან მომხმარებელი.
* **BadRequestException (400 Bad Request)**: არასწორად გამოგზავნილი მონაცემების ან ვალიდაციის დარღვევის შემთხვევაში.

## მონაცემთა მოდელები და ბიზნეს წესები

### 1. მომხმარებლები და უსაფრთხოება
`User` კლასი ახორციელებს `UserDetails` ინტერფეისს Spring Security-სთან ინტეგრაციისთვის. მომხმარებელს გააჩნია კონკრეტული როლი (`Role`), რომელიც განსაზღვრავს მის უფლებებს (Authorities). წვდომები შეზღუდულია ანოტაციით `@PreAuthorize`. მაგალითად:
* პროდუქტის სანახავად საჭიროა `product:read` უფლება.
* პროდუქტის დასამატებლად ან რედაქტირებისთვის საჭიროა `product:add` (ადმინისტრატორის უფლება).

### 2. პროდუქციის კატალოგი და დინამიური ძებნა
`ProductRepository`-ში გაწერილია კომპლექსური JPQL მოთხოვნა (`@Query`), რომელიც უზრუნველყოფს პროდუქტების მოქნილ ფილტრაციას:
* ძებნა არის Case-Insensitive (ტექსტი გადაყავს Lowercase-ში) და ეძებს ნაწილობრივ დამთხვევას დასახელებაში.
* ფილტრაცია ითვალისწინებს ფასის მინიმალურ (`priceFrom`) და მაქსიმალურ (`priceTo`) ზღვრებს.
* ყველა პარამეტრი არასავალდებულოა (`IS NULL` შემოწმების წყალობით) და მონაცემები ბრუნდება პაგინირებული სახით (`Pageable`).

### 3. საყიდლების კალათა (Cart Cycle)
* `Cart` მოდელი `OneToOne` კავშირშია მომხმარებელთან და `OneToMany` კავშირშია თავის ელემენტებთან (`CartItem`).
* გამოყენებულია `cascade = CascadeType.ALL` და `orphanRemoval = true`, რაც ნიშნავს, რომ თუ კალათიდან წაიშლება ელემენტი, ის ავტომატურად წაიშლება ბაზიდანაც.

### 4. შეკვეთები და მარკეტინგი (Orders, Discounts & Coupons)
* **Checkout Engine**: შეკვეთის გაფორმებისას (`/api/orders/checkout`) კალათის მიმდინარე შიგთავსი გადადის `OrderItem` ცხრილში. აქ ფიქსირდება ყიდვის მომენტში არსებული პროდუქტის ფასი, რათა მომავალში ფასის ცვლილებამ გავლენა არ მოახდინოს ძველ შეკვეთებზე.
* **კუპონები**: სისტემა ამოწმებს პრომო კოდის ვალიდურობას `findByCodeAndActiveTrue` მეთოდით და ვადის გასვლის თარიღს (`expiryDate`).
* **ფასდაკლებები**: `DiscountRepository` ფილტრავს აქტიურ ფასდაკლებებს დროის მიხედვით (`findAllByEndDateAfter`). ფასდაკლების დადებისას სისტემა ავტომატურად ითვლის ახალ ფასს.
